# 物流小车多站点往返控制开发文档 v1

## 1. 目标

实现一个基于 4 路循迹传感器的小型物流车控制程序，满足以下行为：

* 小车运行在一条**非环形但可双向往返**的线路上。
* 线路包含 **3 个站点**。
* 启动前，用户通过按钮设置目标站点编号 `targetStation`。
* 小车启动后沿线前进，在**固定目标站点**停车并卸货。
* 卸货完成后继续沿**同一方向**前进。
* 行驶到线路尽头后执行 **180 度掉头**。
* 掉头后进入下一趟运行，并再次在同一个目标站点停车卸货。
* 整体要求是在正常循线条件下可以**持续循环运行**。

---

## 2. 当前设计结论

### 2.1 传感器语义

程序统一采用：

* `Black = 1`
* `White = 0`

### 2.2 线路结构

线路由以下几类标记组成：

1. **正常循迹主线**
   小车平时沿主线循迹，典型居中读数为 `0110`。

2. **站点中心标记（station mark）**
   站点中心有一段横向黑胶带，经过时四个传感器都会读到黑色，即 `1111`。

3. **站点缺口标记（gap marks）**
   在站点中心前后，对称布置若干主线断点。小车经过时会短暂读到 `0000`。

4. **线路尽头**
   主线直接结束，不再恢复。小车会从正常循线状态进入持续的 `0000`。

### 2.3 站点编码规则

按设计：

* 第 1 站：`b = 0`，因此 `currentStation = b + 1 = 1`
* 第 2 站：站点前会先经过 1 个缺口，因此进入 `1111` 时 `b = 1`，故 `currentStation = 2`
* 第 3 站：站点前会先经过 2 个缺口，因此进入 `1111` 时 `b = 2`，故 `currentStation = 3`

公式固定为：

`currentStation = b + 1`

并且**只在第一次确认进入 `1111` 时计算**。

## 3. 硬件交互

### 3.1 按钮

使用两个按钮：

* **按钮 A：目标站设置按钮**

  * 每按一次，`targetStation++`
  * 上限封顶为 `3`
  * 超过 `3` 不再增加

* **按钮 B：确认并启动按钮**

  * 确认当前目标站点
  * 启动小车运行

### 3.2 初始摆放约束

启动前，人工将机器人摆放在正常主线上。
按下“确认并启动”按钮时，传感器读数应处于正常循线状态，例如 `0110`。

---

## 4. 变量设计

变量名允许 Codex 自行重命名，但逻辑含义应保持一致。

### 4.1 必需变量

* `targetStation`
  用户预设目标站点，取值范围 `1 ~ 3`

* `stationNum`（简称 `s`）
  当前这一趟已经通过的站点数
  本文中它与 `passedStationCount` 是**同一个变量**

* `boxNum`（简称 `b`）
  当前站点上下文中的缺口计数

* `mode`
  可选第三个状态量，用于区分当前缺口处理方向
  建议取值：

  * `ADD`：遇到有效 `0000` 时执行 `b++`
  * `SUB`：遇到有效 `0000` 时执行 `b--`

### 4.2 建议变量

* `isTurning`

  * 是否处于转弯 / 掉头状态
  * 为 `true` 时，完全屏蔽站点 / 缺口 / 尽头检测

* `isDropping`

  * 是否正在执行卸货动作

* `lastButtonTime` / `lastSensorEventTime`

  * 用于按钮或事件防抖

---

## 5. 初始化规则

程序启动后，在正式运行前：

* `s = 0`
* `b = 0`
* `mode = ADD`
* `isTurning = false`
* `isDropping = false`

掉头完成并重新开始下一趟时，至少重置：

* `s = 0`
* 建议同时显式重置：

  * `b = 0`
  * `mode = ADD`

> 注：这一点虽然之前只明确要求 `s` 清零，但实现时建议连同 `b` 和 `mode` 一起复位，避免脏状态进入下一趟。

---

## 6. 事件定义

### 6.1 正常循线事件

当传感器处于正常循线模式时，执行已有循线逻辑。

已知旧代码中已有正常循线逻辑，可沿用；只需将黑白判断改为：

* 黑 = 1
* 白 = 0

### 6.2 缺口事件（gap event）

当传感器检测到 `0000` 时：

* **立刻计数一次**
* 然后执行 `delay(100)` 屏蔽重复触发
* `100ms` 后期望小车已运行到缺口后方主线上

根据当前 `mode`：

* 若 `mode == ADD`，则执行 `b++`
* 若 `mode == SUB`，则执行 `b--`

### 6.3 站点事件（station event）

当传感器**第一次进入** `1111` 时：

* 立刻认定为一次站点事件
* 执行 `delay(100)` 作为屏蔽 / 防抖
* 进入站点事件处理流程

站点事件处理规则：

1. 计算当前站点编号：`currentStation = b + 1`
2. 执行 `s++`
3. 若 `currentStation == targetStation`：

   * 停车
   * 执行卸货
   * 等待舵机回位完成
   * 恢复前进
4. 站点事件结束后，缺口处理模式将进入“站后减计数逻辑”

### 6.4 完整站点通过事件

当满足以下条件时，判定“完整经过了一个站点标记”：

* 已经发生过一次站点事件（进入过 `1111`）
* 当前处于 `SUB` 模式
* 后续检测缺口使 `b` 重新减回 `0`

此时：

* 该事件视为一个**瞬时事件**
* **立刻**切回 `mode = ADD`
* 小车继续正常前进

### 6.5 尽头事件（end event）

当前设想为：

* 当 `s == 3 && b == 0` 时，说明本方向已完整经过三个站点
* 若此后再次遇到 `0000`
* 并且 `0000` 持续超过阈值 `300ms`
* 则判定为线路尽头
* 执行掉头

> 注：此处与普通缺口 `0000` 在逻辑上存在优先级问题，见第 11 节“未完全定死事项”。

---

## 7. 模式切换规则

### 7.1 初始模式

* 启动时：`mode = ADD`
* 每次掉头重新开始新一趟时：建议 `mode = ADD`

### 7.2 站点前后模式规则

#### 站点前

在正常情况下，小车处于 `ADD` 模式：

* 每遇到一个缺口，执行 `b++`

#### 进入站点中心

当第一次进入 `1111` 并完成站点事件后：

* 切换到 `SUB` 模式

#### 站点后

在 `SUB` 模式下：

* 每遇到一个缺口，执行 `b--`

当 `b == 0` 时：

* 视为当前站点完整走完
* 立刻切回 `ADD`

---

## 8. 去程 / 回程统一逻辑

本方案不是“按来回方向切换不同目标站顺序”，而是：

* **每一趟都在同一个目标站停下卸货**
* 掉头后 `s` 清零，重新开始新一趟
* 同样按站点物理标记重新识别第 1 / 2 / 3 站

也就是说：

* 目标站点编号是固定的
* 不是按“本趟顺序中的第几个站”决定停车
* 而是按当前站点物理编号决定停车

---

## 9. 掉头逻辑

### 9.1 触发条件

掉头前提：

* `s == 3`
* `b == 0`
* 之后再次检测到 `0000`
* 并且 `0000` 持续超过 `300ms`

则执行掉头。

### 9.2 掉头执行方式

第一版允许沿用旧代码方案：

* 使用电机转向逻辑执行 180 度掉头
* 目前可继续采用 `delay(...)` 方式粗略控制转角

### 9.3 掉头期间规则

一旦进入掉头状态：

* `isTurning = true`
* **完全屏蔽**以下检测：

  * 站点检测
  * 缺口检测
  * 尽头检测

掉头完成后：

* `isTurning = false`
* 重置新一趟相关变量
* 重新开始循线

### 9.4 掉头后找线

该部分允许两种策略：

1. 若旧代码已有稳定的“掉头后找线”逻辑，则优先沿用
2. 若 Codex 评估旧逻辑不适合，则允许自行设计新的找线方案

异常情况下，默认策略为：

* **继续找线**
* 不主动停车报错

---

## 10. 目标站卸货逻辑

### 10.1 到站判断

只在站点事件触发时判断：

* `currentStation = b + 1`
* 若 `currentStation == targetStation`

  * 则执行停车卸货

### 10.2 卸货时序

到目标站后：

1. 停车
2. 执行卸货舵机动作
3. 等待舵机回位完成
4. 恢复前进

### 10.3 重复卸货规则

当前设计要求：

* 一个方向算一趟
* 掉头后重新进入下一趟
* 每一趟都在固定目标站点停车并卸货
* 即允许在往返循环中**重复在同一物理站点卸货**

---

## 11. 尚未完全定死 / Codex 需要显式评估的事项

以下内容尚未被最终形式化，但第一版开发时必须由 Codex 显式处理或给出实现建议。

### 11.1 普通缺口与尽头的 `0000` 冲突

当前规则为：

* 任何时候检测到 `0000` 先计数并 `delay(100)`
* 但在 `s == 3 && b == 0` 的条件下，若 `0000` 持续 `300ms` 又要判为尽头

这两者在第三站之后第一次遇到尽头时会冲突。
Codex 需要明确：

* 是先把它当作尽头候选，再决定是否计数
* 还是先计数，再回滚
* 或者进入专门的 `END_CANDIDATE` 状态

### 11.2 `0000` 与 `1111` 的防抖实现是否足够稳

当前只要求第一版使用：

* 检测即触发
* 然后 `delay(100)` 屏蔽重复触发

这是可行的初版方案，但不一定最稳。
Codex 可以在不偏离需求的前提下给出更稳妥实现建议。

### 11.3 尽头 300ms 的唯一实现方式

目前只给出阈值，没有限定唯一实现。
Codex 需要决定具体方法，例如：

* 连续采样法
* `millis()` 计时法
* 进入 end-candidate 状态后前探确认

### 11.4 掉头后找线的细节

允许沿用旧代码，也允许重构。
Codex 应评估原实现是否适配新状态机。

### 11.5 异常恢复策略

目前要求为：

* 异常时继续找线

但未定义最大尝试时间 / 次数。
第一版可先不设上限，后续再测试补充。

## 12. 推荐的程序结构

建议 Codex 使用**状态机 + 旧循线逻辑复用**的方式开发，而不是继续堆叠 if / else。

建议至少包含以下状态：

* `SETTING_TARGET`：设置目标站点
* `RUNNING`：正常循线与事件识别
* `DROPPING`：目标站停车卸货
* `TURNING`：掉头
* `SEEKING_LINE`：掉头后重新找线（若需要单独拆分）

其中：

* `TURNING` 状态必须屏蔽全部站点 / 缺口 / 尽头检测
* `RUNNING` 状态中再根据 `mode` 处理 `b++ / b--`

## 15. 关键事件示例

### 15.1 第 2 站（去程）典型传感器过程

* `0110`：正常循线
* `0000`：检测到站前第 1 个缺口，`mode = ADD`，执行 `b++`
* `0110`：恢复正常循线
* `1111`：检测到站点中心，计算 `currentStation = b + 1 = 2`，执行 `s++`
* `0110`：继续前进
* `0000`：检测到站后对称缺口，`mode = SUB`，执行 `b--`
* `0110`：恢复正常循线，且 `b == 0`，切回 `mode = ADD`

### 15.2 第 1 站（去程）典型传感器过程

* `0110`：正常循线
* `1111`：站点中心，因 `b == 0`，故 `currentStation = 1`
* `s++`
* 若目标站为 1，则停车卸货
* 继续前进后进入站后缺口处理

## 16. 测试建议

第一版完成后，建议重点测试：

1. 按钮加站功能是否封顶在 3
2. 第 1 站是否可稳定识别
3. 第 2 / 3 站前后的 `b++ / b--` 是否符合预期
4. 目标站是否每趟都能停车卸货
5. 第三站后是否会把尽头误判为普通缺口
6. 掉头后是否确实进入新一趟，并从 `s = 0` 重新计数
7. 转弯状态下是否完全屏蔽其他检测
8. `delay(100)` 和尽头 `300ms` 阈值是否需要微调

## 17. 本次实现落地说明

以下内容已按当前代码版本落地，便于后续接线和调参时直接对照：

### 17.1 当前代码引脚约定

* 按钮 A（目标站设置）使用 `D12`
* 按钮 B（确认启动）使用 `D6`
* 两个按钮均采用 `INPUT_PULLUP`

> 若后续硬件接线不同，只需修改程序顶部的 `PIN_BTN_SELECT` / `PIN_BTN_START` 宏定义即可。

### 17.2 当前代码的防抖与事件去重

* 按钮防抖采用 `millis()` 边沿去抖，稳定按下超过 `30ms` 才算一次有效按键
* `0000` 与 `1111` 都采用“首次进入触发一次 + `delay(100)` 屏蔽重复触发”的实现
* 因此长按按钮不会连续加站，同一段站点标记 / 缺口也不会在一次通过中重复计数

### 17.3 当前代码如何区分普通缺口与尽头

当前实现选择：

* 只有在 `s == 3 && b == 0 && mode == ADD` 时，`0000` 才先被视为“尽头候选”
* 若 `0000` 连续持续超过 `300ms`，则判定为尽头并执行掉头
* 若中途恢复到非 `0000`，则放弃尽头判定，并把这次 `0000` 当作普通缺口事件处理

这样可以避免“先按缺口计数、再回滚”的状态污染。

### 17.4 第 1 站的模式切换补充

文档原规则中，第 1 站进入站点时 `b == 0`，但站后没有对称缺口可再把 `b` 减回 `0`。

因此当前代码补充了一个必要边界条件：

* 站点事件结束后先切到 `SUB`
* 若此时 `b` 已经是 `0`，则立刻判定“该站已完整通过”
* 立即切回 `ADD`

这样可以避免第 1 站通过后仍停留在 `SUB`，从而把第 2 站前缺口误计成 `b--`。
